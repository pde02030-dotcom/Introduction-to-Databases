### 2. 3단계 데이터베이스 구조 (The Three-Schema Architecture)

| 슬라이드 | 제목 | 내용 구성 |
| :--- | :--- | :--- |
| **S 2.1** | **3-스키마 아키텍처 개요** | * **핵심:** 데이터의 **논리적/물리적 정의**를 분리하여 시스템의 유연성을 극대화한 구조.  |
| **S 2.2** | **1. 외부 스키마 (External Schema) / 서브 스키마** | * **관점:** **사용자(End-User) 또는 응용 프로그래머의 관점.** |
| | | * **특징:** 전체 DB 중 **자신에게 필요한 논리적 부분**만 정의하며, 나머지는 은폐(Hiding)한다. SQL의 **View** 개념과 밀접하게 연관됨. |
| **S 2.3** | **2. 개념 스키마 (Conceptual Schema) / 전체 스키마** | * **관점:** **조직 전체의 관점.** **DBA**가 관리하는 DB의 유일한 논리적 구조. |
| | | * **역할:** 모든 개체, 속성, 관계, 무결성 제약 조건(PK, FK 등)을 **논리적**으로 정의. 저장 구조나 접근 경로는 일절 언급하지 않음. |
| **S 2.4** | **3. 내부 스키마 (Internal Schema) / 물리적 스키마** | * **관점:** **데이터가 실제로 저장되는 방식의 관점.** |
| | | * **역할:** 데이터 레코드의 **물리적 저장 구조**, 인덱스(Index)의 생성, 저장 경로, 접근 방법 등을 정의. 성능(Performance) 최적화와 직결됨. |

### 3. 스키마와 인스턴스 (Schema vs Instance)

| 슬라이드 | 제목 | 내용 구성 |
| :--- | :--- | :--- |
| **S 3.1** | **스키마 (Schema)의 정적 특성** | * **정의:** 데이터베이스의 **구조(뼈대)**와 **제약 조건**에 대한 정의. |
| | | * **특성:** DB가 구축된 후에는 **쉽게 변하지 않는** 정적이고 영구적인 구조. (비유: 아파트 설계 도면) |
| **S 3.2** | **인스턴스 (Instance)의 동적 특성** | * **정의:** 특정 시점에 데이터베이스에 **실제로 저장된 데이터 집합.** |
| | | * **특성:** `INSERT`, `DELETE`, `UPDATE`에 따라 **계속 변화하는** 동적인 상태. (비유: 설계 도면에 따라 지어진 아파트에 실제 입주한 사람들의 현황) |

### 4. 사상 (Mapping)과 데이터 독립성

| 슬라이드 | 제목 | 내용 구성 |
| :--- | :--- | :--- |
| **S 4.1** | **사상 (Mapping)의 역할** | * **정의:** 서로 다른 스키마 간의 **변환 규칙**. DBMS가 데이터 요청을 처리하기 위해 필요. |
| | | * **두 가지 사상:** **외부/개념 사상(External/Conceptual Mapping)**과 **개념/내부 사상(Conceptual/Internal Mapping)**. |
| **S 4.2** | **데이터 독립성의 정의** | * **개념:** 스키마의 한 단계가 변경되어도 **바로 위 단계의 스키마에 영향을 미치지 않도록** 함으로써 응용 프로그램의 수정 없이 DB를 변경할 수 있는 성질. |
| **S 4.3** | **1. 논리적 데이터 독립성 (Logical Data Independence)** | * **정의:** **개념 스키마**가 변경되어도 **외부 스키마**에 영향을 주지 않는 성질. |
| | | * **예시:** 개념 스키마에서 새로운 속성 추가/기존 속성 삭제 시, 기존 응용 프로그램(외부 스키마)은 **사상**을 통해 그대로 유지됨. (DBMS가 변경된 부분을 중재) |
| **S 4.4** | **2. 물리적 데이터 독립성 (Physical Data Independence)** | * **정의:** **내부 스키마**가 변경되어도 **개념 스키마**에 영향을 주지 않는 성질. |
| | | * **예시:** 저장 장치 교체, 인덱스 추가/제거, 파일 저장 방식 변경 등 물리적 변경 시 개념 스키마(그리고 외부 스키마)는 영향받지 않음. **성능 최적화**를 위한 필수 조건. |

### 5. 데이터 사전 (Data Dictionary)

| 슬라이드 | 제목 | 내용 구성 |
| :--- | :--- | :--- |
| **S 5.1** | **데이터 사전 (Data Dictionary)의 역할** | * **정의:** **시스템 카탈로그**라고도 불림. DB 내의 모든 스키마, 제약 조건, 무결성 규칙, 사용자 정보 등 **메타데이터(Metadata)**를 저장하는 곳. |
| | | * **특징:** DBMS가 스스로 유지하고 관리하는 핵심 시스템 테이블. (DBMS의 두뇌) |
| **S 5.2** | **데이터 사전의 활용** | * DBMS는 모든 질의(Query) 처리 시 데이터 사전을 참조하여 유효성과 접근 권한을 검사한다. **DBMS의 '자체 관리' 능력**의 근간. |
