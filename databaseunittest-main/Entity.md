# 🔶 엔티티(Entity)

### ✔ **정의**

엔티티(Entity)는

> **현실 세계에서 독립적으로 존재하며, 데이터를 저장할 필요가 있는 개념적 대상(thing)**
> 을 의미한다.

즉, **저장해야 할 정보의 집합**이며
ERD(엔터티-관계 다이어그램)의 가장 핵심 요소이다.

---

# 🔶 엔티티의 특징 (반드시 충족해야 함)

현업 데이터 모델러들은 아래 “5가지 기준”으로 엔티티를 판단합니다:

1. **업무에서 반드시 필요하다**
   → 저장하거나 관리할 이유가 있어야 한다.

2. **고유하게 식별 가능하다(PK 존재)**
   → PK(Primary Key) 없이 엔티티가 될 수 없다.

3. **2개 이상의 인스턴스(데이터 행)가 존재한다**
   → “집합”이어야 한다. 단일 데이터는 엔터티가 아니다.

4. **속성(Attribute)을 2개 이상 가진다**
   → 최소한 2개 이상의 컬럼을 가져야 한다.

5. **다른 엔터티와 관계(Relationship)가 존재하거나 존재 가능하다**
   → 완전히 고립된 데이터는 엔터티로 보기 어렵다.

---

# 🔶 엔티티 예시

| 엔티티            | 설명                      |
| -------------- | ----------------------- |
| **Student**    | 학생 정보                   |
| **Course**     | 강의 정보                   |
| **Enrollment** | 학생-강의 수강 관계 (N:M 관계 해결) |
| **Order**      | 주문                      |
| **OrderItem**  | 주문의 상세 상품               |
| **Member**     | 회원                      |
| **Product**    | 상품                      |

---

# 🔶 엔티티 vs 테이블 (중요)

**엔티티는 모델링 단계**
**테이블은 실제 구현 단계**

| 엔티티                 | 테이블                        |
| ------------------- | -------------------------- |
| 개념적·논리적 모델          | 물리적 모델                     |
| DBMS 독립적            | DBMS 의존적(MySQL/Oracle 등)   |
| PK 후보 기술            | PK 실제 생성                   |
| 속성 정의 중심            | 데이터 타입 / 인덱스 / 제약조건 포함     |
| 관계(Relationship) 중심 | FK, ON DELETE 등 실제 제약조건 적용 |

즉, **엔티티 Entity → 테이블 Table** 로 변환된다.

---

# 🔶 엔티티의 구성 요소

### 1) **엔터티명(Entity Name)**

* 명사형
* 단수형 사용 권장
  예: Student, Member, Order

### 2) **속성(Attribute)**

→ 컬럼 구성 요소
예: student_id, name, birth_date

### 3) **식별자(Identifier, PK)**

→ 유일성 확보
예: student_id

### 4) **관계(Relationship)**

→ 다른 엔티티와의 연결
예: Student 1:N Enrollment

---

# 🔶 엔티티의 종류

### ✔ 1) 강엔터티(Strong Entity)

PK를 스스로 갖고 있고
다른 엔티티에 의존하지 않음.

예: Student, Product

---

### ✔ 2) 약엔터티(Weak Entity)

부모 엔티티의 PK를 포함한 **복합 PK**로만 식별 가능한 엔티티.

예: OrderItem

* order_id(PK+FK)
* product_id(PK+FK)

부모 없이 존재할 수 없음 → 식별관계 사용

---

### ✔ 3) 개념적 엔티티 (업무 중심)

“고객”“계약”“과정”“주문” 등
업무에서 나타나는 개념.

---

### ✔ 4) 관계 엔티티(Association Entity)

N:M 관계를 1:N + N:1로 풀기 위해 만든 엔티티.
예: Enrollment(수강), OrderItem(주문상세) 등.

---

# 🔶 엔티티 판단 예시

### 예: “로그인 기록(LoginHistory)”는 엔티티일까?

✔ 데이터가 여러 건 저장됨
✔ 고유 ID 존재(login_id)
✔ Member와 관계 있음
✔ 속성 다수(timestamp, ip, user_agent)

→ **엔티티 맞음**

---

### 예: “주소(Address)”는 엔티티인가?

* Member가 주소를 여러 개 가질 수 있음
* 주소는 속성이 많음
* ID 부여 가능

→ **엔티티**
(보통 Address 엔티티를 만들고 Member와 1:N 관계)

---

### 예: “전화번호 하나만 저장하는 단일 필드”는 엔티티인가?

→ **아님**
그냥 Member 엔터티의 속성일 뿐.

---

# 🔶 엔티티 도출(Entitiy Extraction) 기준

1. **업무 규칙(Business Rule)** 존재
2. **독립적으로 저장할 필요성**
3. **업무 프로세스에서 반복적으로 언급**
4. **정규화 과정에서 자연스럽게 분리됨**
5. **다른 엔티티에 의해 참조됨 or 참조 가능성**

---

# 🔶 최종 정리

> **엔티티(Entity)는 업무에서 독립적으로 존재하며, 저장해야 할 데이터 집합을 의미하고, 속성과 식별자를 갖는 개념적 단위이다.
> 엔티티는 데이터 모델링(ERD)의 핵심이며, 최종적으로 테이블로 구현된다.**
