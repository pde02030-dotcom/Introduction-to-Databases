## 🗃️ Chapter 05. 관계 데이터 모델 (Relational Data Model)

---

### 1. 학습 목표 및 관계 모델의 수학적 기반

| 슬라이드 | 제목 | 내용 구성 |
| :--- | :--- | :--- |
| **S 1.1** | **챕터 5 학습 목표 및 용어** | 📌 **목표:** 관계 모델의 기본 용어와 구조를 수학적 정의 기반으로 이해하고, 데이터의 정확성을 보장하는 **개체/참조 무결성** 제약 조건을 완벽하게 설명할 수 있다. |
| | | * **주요 용어:** 릴레이션, 튜플, 속성, 도메인, 차수(Degree), 카디널리티(Cardinality), 후보키, 기본키, 무결성 제약 조건. |
| **S 1.2** | **관계 모델의 등장 배경** | * **배경:** 1970년 Edgar F. Codd가 제안. **수학적인 집합론(Set Theory)**과 **술어 논리(Predicate Logic)**에 기반하여 모델의 견고함 제공. |
| | | * **철학:** 데이터의 **논리적 구조**와 **물리적 구현**을 완전히 분리 (**물리적 독립성** 보장). |
| **S 1.3** | **기본 정의: 도메인과 릴레이션** | * **도메인 (Domain):** 속성이 가질 수 있는 **모든 가능한 값의 집합.** (예: '나이' 속성의 도메인은 $\{1 \le x \le 120\}$) |
| | | * **릴레이션 (Relation):** 하나 이상의 도메인 $D_1 \times D_2 \times \dots \times D_n$ 의 **카티션 곱(Cartesian Product)**의 부분 집합. $\rightarrow$ 즉, 수학적 정의에 기반한 테이블. |

### 2. 관계 모델의 구조적 용어

| 슬라이드 | 제목 | 내용 구성 |
| :--- | :--- | :--- |
| **S 2.1** | **릴레이션 (Relation)의 구성 요소** | * **릴레이션:** 테이블. |
| | | * **튜플 (Tuple):** 행(Row). 릴레이션의 인스턴스를 구성하는 원소. |
| | | * **속성 (Attribute):** 열(Column). 튜플이 갖는 특성. |
| **S 2.2** | **릴레이션의 특징** | * **튜플의 유일성:** 모든 튜플은 달라야 함. |
| | | * **튜플의 무순서성:** 튜플의 순서는 중요하지 않음. |
| | | * **속성 값의 원자성:** 모든 속성 값은 더 이상 분해할 수 없는 **원자 값**이어야 함 (**1NF의 전제**). |
| **S 2.3** | **구조적 측정 요소 (차수 vs 카디널리티)** | * **차수 (Degree, $n$):** **속성(컬럼)의 개수.** 릴레이션 스키마의 정의 시 결정되는 **정적인 값.** |
| | | * **카디널리티 (Cardinality, $m$):** **튜플(행)의 개수.** DB의 상태에 따라 변하는 **동적인 값.** |

### 3. 키(Key)의 계층적 구조와 정의

| 슬라이드 | 제목 | 내용 구성 |
| :--- | :--- | :--- |
| **S 3.1** | **키 (Key)의 정의 및 역할** | * **역할:** 튜플을 **고유하게 식별**하거나, 다른 릴레이션과의 **관계를 연결**하는 데 사용되는 하나 또는 복수의 속성. |
| **S 3.2** | **슈퍼키 (Superkey)** | * **정의:** 릴레이션 내에서 **튜플을 유일하게 식별**할 수 있는 속성 또는 속성들의 집합. (불필요한 속성이 포함되어도 무방). |
| **S 3.3** | **후보키 (Candidate Key)** | * **정의:** **유일성(Uniqueness)**과 **최소성(Minimality)**을 모두 만족하는 슈퍼키. $\rightarrow$ **튜플 식별의 최소 단위.** |
| | | * **최소성:** 후보키를 구성하는 어떤 속성 하나라도 제외하면 유일성을 잃는 성질. |
| **S 3.4** | **기본키 (Primary Key, PK)** | * **정의:** 여러 후보키 중 **DBA가 최종적으로 선택**하여 튜플을 대표하고 식별하도록 지정한 키. (하나의 릴레이션에 오직 하나). |
| **S 3.5** | **대체키 (Alternate Key) 및 외래키 (Foreign Key, FK)** | * **대체키:** 후보키 중 PK로 선택되지 않은 나머지 키. |
| | | * **외래키:** 다른 릴레이션의 **PK를 참조**하여 관계를 맺는 속성. 데이터 무결성을 위한 **'릴레이션 간의 연결 고리'** 역할. |

### 4. 무결성 제약 조건 (Integrity Constraints)

| 슬라이드 | 제목 | 내용 구성 |
| :--- | :--- | :--- |
| **S 4.1** | **무결성 (Integrity)의 개념** | * **정의:** 데이터베이스에 저장된 데이터의 **정확성, 일관성, 유효성**을 유지하는 것. DBMS의 가장 중요한 임무 중 하나. |
| **S 4.2** | **1. 개체 무결성 제약 조건 (Entity Integrity Constraint)** | * **적용 대상:** **기본키 (PK)** |
| | | * **규칙:** **릴레이션의 기본키는 NULL 값을 가질 수 없으며, 중복된 값을 가질 수 없다.** |
| | | * **논리:** PK가 NULL이거나 중복되면 튜플을 유일하게 식별할 수 없게 되므로, 데이터베이스의 근간이 무너짐. |
| **S 4.3** | **2. 참조 무결성 제약 조건 (Referential Integrity Constraint)** | * **적용 대상:** **외래키 (FK)** |
| | | * **규칙:** **외래키 값은 참조하는 릴레이션의 기본키 값(참조 범위) 중 하나이거나, NULL 값을 가져야 한다.** |
| | | * **논리:** 존재하지 않는 개체를 참조하는 '허상(Dangling Pointer)' 참조를 방지. 관계의 유효성 유지. |
| **S 4.4** | **참조 무결성 위반 시 처리 방법 (Action)** | * **$\text{ON DELETE/UPDATE}$ 규칙:** DBMS가 무결성 위반 시 취할 조치 정의. |
| | | 1. **CASCADE:** 참조 데이터도 연쇄적으로 삭제/갱신. |
| | | 2. **SET NULL:** 외래키 값을 NULL로 변경. |
| | | 3. **NO ACTION/RESTRICT:** 삭제/갱신 작업을 거부. |
