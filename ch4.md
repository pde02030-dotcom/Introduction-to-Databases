### 2. 개념적 모델링: E-R 모델 심화

| 슬라이드 | 제목 | 내용 구성 |
| :--- | :--- | :--- |
| **S 2.1** | **E-R 모델 (Entity-Relationship Model) 개요** | * **E-R:** 개체, 관계, 속성을 중심으로 현실 세계를 표현하는 개념적 데이터 모델. Peter Chen에 의해 1976년에 제안됨. |
| **S 2.2** | **개체와 속성 (Entity & Attribute)** | * **개체:** 의미 있는 정보 단위. DB 내에서 **튜플(Tuple)**로 변환될 대상. |
| | | * **속성 종류 심화:** **단순 속성**, **복합 속성** (Composite, 예: 주소 $\rightarrow$ 시/구/동), **다치 속성** (Multivalued, 예: 전화번호 목록). |
| | | * **도출 속성 (Derived Attribute):** 다른 속성으로부터 계산 가능한 속성 (예: 생년월일 $\rightarrow$ 나이). |
| **S 2.3** | **관계의 유형과 참여 제약 조건** | * **관계 카디널리티 (Cardinality):** 1:1, 1:N, N:M. (가장 중요) |
| | | * **참여 제약 조건 (Participation Constraint):** |
| | | 1. **전체 참여 (Total Participation):** 모든 개체가 관계에 **필수적**으로 참여. (이중선으로 표기). |
| | | 2. **부분 참여 (Partial Participation):** 일부 개체만 관계에 참여. (단일선으로 표기). |
| **S 2.4** | **약한 개체와 강한 개체 (Weak vs Strong Entity)** | * **약한 개체:** 자신의 PK를 구성하기 위해 **강한 개체의 PK를 의존**해야 하는 개체. (예: 보험 가입자의 '부양가족'). |
| | | * **표기:** 이중 사각형, 관계 마름모도 이중선. |
| **S 2.5** | **E-R 모델의 추상화 기법 (Abstraction)** | * **1. 일반화/특수화 (Generalization/Specialization):** 공통된 개체를 묶어 상위 개체(일반화), 상위 개체를 세분화(특수화). (**ISA 관계**) |
| | | * **2. 집단화 (Aggregation):** 관계(Relationship) 자체를 새로운 하나의 **상위 개체 타입**으로 간주. 복잡한 관계 모델링 시 유용. |

### 3. 논리적 설계: 관계 모델로의 변환 (Mapping)

| 슬라이드 | 제목 | 내용 구성 |
| :--- | :--- | :--- |
| **S 3.1** | **논리적 설계의 목표** | * **목표:** E-R 다이어그램을 특정 DBMS가 지원하는 **관계 모델(테이블)**로 변환. |
| | | * **단계:** E-R 모델 $\rightarrow$ 정규화된 릴레이션 스키마 생성. |
| **S 3.2** | **변환 규칙 1: 개체 타입 및 속성** | * **강한 개체:** 하나의 릴레이션(테이블)으로 변환. |
| | | * **복합 속성:** 단순 속성으로 분해하여 릴레이션의 컬럼이 됨. |
| | | * **다치 속성:** **별도의 독립된 릴레이션**으로 변환되어, 원래 개체의 PK를 **외래키(FK)**로 가짐. |
| **S 3.3** | **변환 규칙 2: 1:1 관계** | * **변환:** 한쪽 릴레이션의 **PK**를 다른 쪽 릴레이션에 **FK**로 추가. (전체 참여 쪽에 추가하는 것이 효율적). |
| **S 3.4** | **변환 규칙 3: 1:N 관계** | * **변환:** 1인 쪽 릴레이션의 **PK**를 N인 쪽 릴레이션에 **FK**로 추가. (가장 일반적인 관계 변환). |
| **S 3.5** | **변환 규칙 4: N:M 관계** | * **변환:** N:M 관계는 **별도의 새로운 릴레이션**으로 변환됨. |
| | | * **새로운 릴레이션의 PK:** 원래 두 개체의 **PK를 조합**하여 복합키(Composite Key)로 사용. 이 두 키는 각각 원래 개체를 참조하는 **FK**가 됨. |
| **S 3.6** | **변환 규칙 5: 약한 개체** | * **변환:** 약한 개체는 독립된 릴레이션으로 변환되지만, **PK**는 **강한 개체의 PK (FK 역할) + 약한 개체의 식별 속성**을 합친 복합키가 됨. |

### 4. 물리적 설계의 관점 소개

| 슬라이드 | 제목 | 내용 구성 |
| :--- | :--- | :--- |
| **S 4.1** | **논리적 설계와 물리적 설계의 분리** | * **논리적:** 데이터를 어떻게 **구조화**할 것인가. (DBMS 독립적) |
| | | * **물리적:** 데이터를 컴퓨터에 어떻게 **저장**할 것인가. (DBMS 종속적) |
| **S 4.2** | **물리적 설계의 주요 고려 사항** | * **성능(Performance)**에 초점: **인덱스 전략** (어떤 컬럼에 인덱스를 생성할지), **저장 공간 할당**, **클러스터링** (관련 데이터를 물리적으로 가깝게 저장) 등. |
| | | * **DBA의 역할:** SQL 쿼리 처리 속도를 최적화하기 위한 마지막 단계. |
